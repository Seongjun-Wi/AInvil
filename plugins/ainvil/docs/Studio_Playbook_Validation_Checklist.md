# Studio Playbook Validation Checklist

Use this checklist to verify that recommendations, reviews, and production decisions follow the AInvil Studio Playbook.

## 1. Creative Ownership

| check | pass/fail | evidence |
| --- | --- | --- |
| User creative ownership is preserved. |  |  |
| Player-facing changes requiring confirmation are marked appropriately. |  |  |
| Prototype defaults do not overwrite production intent. |  |  |

## 2. Evidence-Based Decisions

| check | pass/fail | evidence |
| --- | --- | --- |
| Recommendation references Production State Graph, review records, validation evidence, or source documents. |  |  |
| Unknowns are marked instead of guessed. |  |  |
| Confidence is separated from validation level. |  |  |

## 3. Incremental Production

| check | pass/fail | evidence |
| --- | --- | --- |
| Work is scoped to the current milestone or a clear next action. |  |  |
| The increment can be validated. |  |  |
| Placeholders and replacement conditions are tracked. |  |  |

## 4. Review Before Commitment

| check | pass/fail | evidence |
| --- | --- | --- |
| Required review type has been identified. |  |  |
| Review record exists or review is explicitly not required. |  |  |
| Review decisions are not treated as user approval when creative scope changes. |  |  |

## 5. Validation Before Completion

| check | pass/fail | evidence |
| --- | --- | --- |
| Requirements exist. |  |  |
| Acceptance criteria exist. |  |  |
| Validation evidence exists at the required level. |  |  |
| Remaining validation gaps are documented. |  |  |

## 6. Scope Discipline

| check | pass/fail | evidence |
| --- | --- | --- |
| Feature supports the core fantasy or milestone goal. |  |  |
| Feature creep risk has been considered. |  |  |
| Weak features are cut/deferred instead of expanded. |  |  |

## 7. Design Quality

| check | pass/fail | evidence |
| --- | --- | --- |
| Core fantasy contribution is clear. |  |  |
| Core loop contribution is clear. |  |  |
| Player experience impact is clear. |  |  |
| Replayability or progression impact is considered when relevant. |  |  |
| Production cost is visible. |  |  |

## 8. Technical Quality

| check | pass/fail | evidence |
| --- | --- | --- |
| Implementation is maintainable. |  |  |
| Traceability is preserved. |  |  |
| Validation is feasible. |  |  |
| Technical debt is recorded when accepted. |  |  |

## 9. Knowledge Evolution

| check | pass/fail | evidence |
| --- | --- | --- |
| Lessons learned are captured when useful. |  |  |
| Successful patterns are recorded. |  |  |
| Failed approaches or rejected options are preserved when useful. |  |  |
